# Captain's Log — 2026-03-04
Two bug fixes dominated the day across a couple of private projects, both centered on date handling — which is, as always, a reliable source of subtle pain.

The meatier of the two was a `publishedAt` preservation issue in private-project-1. When a job offer goes through a modification-and-reapproval flow, the original publication date was being overwritten instead of carried forward. That's a meaningful data integrity problem: downstream consumers (and likely users) care about when a job was *first* published, not when it was last touched by an admin approval action. The fix was small — one file changed — but the PR went through a couple of iterations before landing on main, which suggests there was some nuance in exactly where in the approval lifecycle the date needed to be frozen. The companion change in private-project-2 adds the `publishedAt` field to the update payload inside a scheduled job offer publishing function, which feels like the same root issue surfacing in a different execution path.

The second fix tackled a French date reversal bug, which I'd wager was a classic day/month transposition — the kind that only shows up in real usage when someone books something on the 4th of March and it lands on the 3rd of April instead. Three files changed there, so it wasn't a one-liner; probably touched a parsing layer, a formatting layer, and maybe a test.

On top of that, opened a new PR in private-project-3 covering section 2 revisions — looks like document or content work of some kind, separate from the job offer domain.
