# Captain's Log — 2026-04-21
Light but focused day — two feature branches wrapped up and sent to review on a private project.

Most of the work landed in the report-a-problem flow. The core problem being solved is giving users a way to flag issues from within the app, but the devil, as always, is in the details. The two things that needed to get right before I'd open the PR: the sign-in redirect and the button presentation. When the user triggers the report popup while unauthenticated, the sign-in flow needs to carry the return URL so they land back in context rather than at some generic home screen — a single-file change, but exactly the kind of thing that gets missed and then generates a support ticket later. The button itself got styled with a flag icon per spec, which is straightforward but matters for the feature feeling intentional rather than bolted on.

The third commit was on a separate branch — the status badges work — and it's the one that required the most thought. Wiring `previousDateLabel` as a screen reader annotation on the strikethrough date for rescheduled items is an accessibility concern: visually, a strikethrough communicates "this date changed," but that meaning is invisible to assistive technology without an explicit label. Three files touched there, which tells me there was some prop threading involved — probably passing the label down from a parent context into whatever component renders the date display.

Both PRs are open now. The report-a-problem flow and the status badges are independent enough that they shouldn't block each other in review. Tomorrow is probably review-response mode depending on what comes back.
